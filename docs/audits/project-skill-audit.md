# Auditoria de coerência — skill `atomic-skills:project` (e relacionadas)

> **Objetivo.** Validar se a skill `project` (router `skills/core/project.md` + assets
> em `skills/shared/project-assets/` + backing `src/`/`scripts/` + schemas + design CANON)
> é **coerente com o que se propõe a fazer**: oferecer um *método organizado de
> especificar e implementar código*. Não é um code-review de bugs pontuais — é uma
> auditoria de **coerência do método**: a promessa bate com o comportamento, o fluxo
> fecha, o vocabulário e o estado são consistentes, e o método é operável na prática.

- **Escopo:** máximo — tudo que a Project toca (núcleo + ciclo de vida + infra de estado + docs/kb + design + consumer).
- **Dimensões:** (1) doc-vs-comportamento · (2) completude do método · (3) consistência interna · (4) fricção de uso.
- **Iniciado:** 2026-07-02
- **Executor:** sessão interativa + **cron resiliente a cada 30min** (retoma do checkpoint abaixo se a sessão bater rate limit).

---

## Como o loop retoma (LEIA PRIMEIRO a cada run do cron)

1. Leia a tabela **Progresso** abaixo. Pegue a PRIMEIRA unidade com status `TODO` (ou reassuma a que estiver `IN_PROGRESS`).
2. Marque-a `IN_PROGRESS` (edite a tabela) antes de começar.
3. Execute a auditoria daquela unidade conforme o `Escopo/Método` da unidade. Anexe achados em **Achados por unidade**.
4. Marque a unidade `DONE` na tabela + preencha `Achados` (contagem) e `Data`.
5. Repita enquanto houver `TODO` e a sessão responder. Se bater rate limit, PARE — o próximo cron retoma daqui.
6. Quando **todas** as unidades estiverem `DONE`, execute a unidade `F1` (síntese) e então marque o cron para remoção (o próprio F1 instrui isso).

**Regra de honestidade:** um achado só entra confirmado se citar `arquivo:linha` e descrever o
descompasso concreto (o que a prose/promessa diz × o que o código/estado faz). Hipóteses vão
marcadas `[HIPÓTESE]` até verificadas. Severidade: `CRÍTICO` (método quebra/engana) ·
`ALTO` (fricção séria ou drift que induz erro) · `MÉDIO` (inconsistência real, contornável) ·
`BAIXO` (cosmético/doc).

Comportamento: nenhum comando seta `promoted:true` ou remove entry de `emerged[]` (grep em emergence/transitions/router = zero writers). ### B3 — Lane de gates/review

**Verdito:** substancialmente coerente — o loop review→gate→close fecha (drift detectado read-only por `detect-completion`, fechado só pelo path verifier-aware `reconcile`/GATE-R2); `verify`/`review` honram o contrato read-only-unless-`--fix`; GATE-R2/R3 back as claims em JS real; precondição `plan/<slug>` do finalize é garantida upstream pelo always-fork no create-plan. Defeitos são doc-vs-behavior + uma inconsistência auto-declarada no ledger de codex-review, não buracos estruturais.

1. **[MÉDIO] `verify` check #8 "review-gate PASS" é satisfeito por review `local`-only que a cadência codex ainda conta como não-revisado** — `skills/shared/project-assets/project-verify.md:94-97`. Comportamento: `phase-done` escreve `reviewGate:{status:passed,mode:local}` para qualquer fase não-destrutiva (`project-transitions.md:154,159`), e nem GATE-R3 (`validate-state.js:491-503`) nem check #8 inspecionam `mode`. Uma fase revisada só-local fica verde permanentemente no `verify`, enquanto o ledger codex (`last-review.json`) reporta "never run/N behind" para os mesmos commits. Impacto: duas superfícies "está revisado?" dão respostas opostas; usuário confiando em `verify [8] PASS` acredita que houve review cross-model quando só houve local same-model.
2. **[MÉDIO] O ledger NDJSON `review-dedup` (`review-ledger.js`) é totalmente construído mas ligado a nada — camada inteira inerte** — `skills/shared/project-assets/project-drift.md:94-114`. Comportamento: o arquivo auto-documenta que o flip de formato está "DEFERRED — not yet live"; os 4 readers ainda parseiam o ponteiro legacy via `jq '.lastReviewedCommit'`, e `readLedger` lê legacy como `[]` (`review-ledger.js:48,54`) → `alreadyReviewed` é sempre false. Impacto: módulo testado backing uma feature que nunca dispara; `review-due` re-revisa toda vez. Correto-mas-morto — a economia prometida não existe até a migração deferida.
3. **[MÉDIO] `review-due` escreve `last-review.json` incondicionalmente em *qualquer* verdict → um `needs_changes` codex com blockers não-aplicados ainda marca a branch "up to date"** — `skills/shared/project-assets/project-drift.md:164-174`. Comportamento: passo 5 atualiza `lastReviewedCommit:<HEAD sha at review start>` "on completion" sem gate em `verdict`/`counts`; um `needs_changes` com 1 blocker carimba o ponteiro, e como os fixes (passo 6) criam novo HEAD, o commit falho lê verde. Impacto: o sinal de cadência pode mostrar verde sobre um commit que falhou o próprio codex review.
4. **[BAIXO] Lista de normalização do `verify --fix` omite `deferred`, contradizendo o enum real** — `project-verify.md:27-29` vs `normalize.js:27,63,65`. Impacto: drift doc-vs-enum menor; status não-reconhecido que o usuário quis como deferred é coagido p/ `pending` (`:65`), não flagged.
5. **[BAIXO] Trigger vermelho da CODEX REVIEW line "≥7 dias OU phase-done rodou desde" é definido mas incomputável do ponteiro que lê** — `project-drift.md:139` vs renderer `:121-132`. Impacto: 2 dos 4 triggers vermelhos nunca disparam; spec over-promete o que o renderer shipado detecta.
6. **[BAIXO] `verify` check #6 aiDeck FAIL recomenda `verify --fix`, que só normaliza e não resolve a maioria dos STATE_ERROR** — `project-verify.md:85-86`. Impacto: usuário segue `--fix`, vê o erro persistir, sem ponteiro in-message para o remédio real (`migrate`/repair gated). Dead-end brando.

### Veredito de coerência

**A skill `project` É coerente com o que se propõe — como método de *especificar e implementar* código — no seu happy path, mas vaza nas bordas de forma concentrada e padronizada, não difusa.** Três achados estruturam o veredito:

1. **O caminho feliz fecha e os hard gates são reais.** A cadeia idea→plan→materialize→implement→done→phase-done→finalize→archive é contínua; a seam SPEC→implement tem gate de admissão que de fato bloqueia task underspecified; GATE-R2/R3 (evidência/review-honesty), o enum strict de gate-status e a máquina de migração (copy-verify-delete, fail-closed, sem renomear slug, 4 remediações adversariais presentes+testadas) são **enforcement real em código, com teste**. A fidelidade ao design CANON é alta. Isto é um método sério, não teatro.

2. **A incoerência mora nos sub-estados que você raramente atinge** — `emerged[]`, `blocked`, `paused`, e a árvore *legacy flat*. É onde transições de saída e readers com fallback nunca foram completados. O método promete uma ladder com backlog drenável e um dual-layout com fallback; ambos têm buracos que só aparecem fora do trilho principal.

3. **Há um gap recorrente entre "a prose diz que a tool ENFORÇA X" e "X na verdade é disciplina que a IA precisa lembrar".** Vários gates anunciados como determinísticos (G2/G6, N-task emit, mutation-kill, SPEC-LINT-em-validate-state, "rollups em toda mudança") são, no código, prose/self-review do LLM. O método às vezes vende disciplina-de-prose como enforcement determinístico.

**Conclusão:** coerência **ALTA no núcleo, com 3 buracos CRÍTICOS reparáveis** que hoje contradizem a promessa "método organizado" — porque levam um usuário disciplinado a um estado que só sai por edição manual (que a própria skill proíbe), ou fazem o caminho lazy primário nascer hard-blocked pelos próprios detectores da skill.
