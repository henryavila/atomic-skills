## Summary
The migration changes the public fork-link readers to inline-only, but leaves existing sidecar state invisible unless some separate caller explicitly runs `migrateSidecarToInline`. That is a data-loss regression for any repository already carrying fork edges in `links.json`.

The inline write/migration paths also do not preserve the old schema validation boundary. Invalid or malformed edge payloads can now be serialized into `plan.md`, and one malformed legacy `spawnedPlans` shape can be transformed into bogus child slugs instead of rejected.

## Findings

### F-001 [critical] migration — src/links-sidecar.js:169-210

**Evidence:**
```js
export function getSpawnedFrom(planDir) {
  return readPlanFm(planDir)?.fm?.spawnedFrom ?? null;
}

/**
 * @param {string} planDir
 * @returns {object} the parent→child map (phaseId → child slugs) aggregated from
 *   the inline `phases[].spawnedPlans`, or {} when none / no readable plan.md
 */
export function getSpawnedPlans(planDir) {
  const p = readPlanFm(planDir);
  if (!p || !Array.isArray(p.fm.phases)) return {};
  const out = {};
  for (const ph of p.fm.phases) {
    if (ph && ph.id && Array.isArray(ph.spawnedPlans) && ph.spawnedPlans.length) {
      out[ph.id] = [...ph.spawnedPlans];
    }
  }
  return out;
}
```

**Claim:** Existing fork edges that still live only in `links.json` become invisible to all callers as soon as these readers are deployed.

**Impact:** Focus/reconcile/archive/cycle-check logic can treat forked plans as unrelated non-fork plans, causing lost parent-child hierarchy, false multi-active drift, skipped resume gates, or duplicate/cyclic fork writes until a manual migration happens.

**Recommendation:** Add a production migration path before switching reads to inline-only, or make these readers fall back to validated sidecar elo when inline fields are absent and then migrate/delete through a controlled command.

**Confidence:** high

### F-002 [major] validation — src/links-sidecar.js:148-160

**Evidence:**
```js
export function setSpawnedFrom(childPlanDir, link) {
  const { plan, phaseId, taskId, mode } = link;
  // Validate the edge at the write boundary (the inline write does not pass
  // through the links.json schema): a bad mode/shape must never reach plan.md.
  if (!plan || !phaseId || (mode !== 'pause' && mode !== 'parallel')) {
    throw new Error(`invalid spawnedFrom edge: requires non-empty plan + phaseId and mode in {pause, parallel} — got ${JSON.stringify(link)}`);
  }
  const edge = { plan, phaseId, mode };
  if (taskId !== undefined) edge.taskId = taskId;
  const p = readPlanFm(childPlanDir);
  if (!p) throw new Error(`cannot set spawnedFrom: no readable plan.md at ${planMdPath(childPlanDir)}`);
  p.fm.spawnedFrom = edge;
  writePlanFm(childPlanDir, p.fm, p.body);
  return p.fm;
}
```

**Claim:** The inline `spawnedFrom` writer only validates truthiness and mode, not the schema-required string/minLength shape for `plan`, `phaseId`, or `taskId`.

**Impact:** Calls such as `{ plan: {}, phaseId: 7, taskId: null, mode: "pause" }` write invalid frontmatter that `plan.schema.json` later rejects; the old sidecar writer would have rejected these at write time.

**Recommendation:** Reuse the same Ajv validation for a normalized edge object before mutating `plan.md`, or add a dedicated inline edge validator that enforces string type and minLength for every field.

**Confidence:** high
