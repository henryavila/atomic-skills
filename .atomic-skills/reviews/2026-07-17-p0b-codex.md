**Findings**

- **Major** [src/install.js:621](/Volumes/External/code/atomic-skills/src/install.js:621): `revertDroppedAutoUpdateEffects()` runs before the driver’s incomplete-transaction guard. The driver only calls `assertNoIncompleteTransaction()` inside `installer.install()` afterward. If the existing manifest is incomplete from a prior crash, this helper can still read it, mutate `.claude/settings.json`, `.grok/hooks/...`, and `.atomic-skills/hooks/version-check.sh`, then the driver rejects the install. That breaks the recovery invariant: disk has been changed outside the incomplete journal the recovery flow is about to trust. The helper should either run after the same incomplete-transaction check or perform that check itself before reading/reverting anything.

- **Major** [src/install.js:621](/Volumes/External/code/atomic-skills/src/install.js:621): the drop-revert runs outside the installer lock. The subsequent driver install acquires locks in `@henryavila/minimalist-installer` only after this helper, but the helper reads the manifest, writes its pending ledger, and reverts files without serialization. A concurrent install can therefore have a journal that claims auto-update effects are present while this helper removes them, especially if the shrink process crashes before reaching `installer.install()`. This needs to be under the same install-root lock as the driver, or implemented inside the engine transaction path.
