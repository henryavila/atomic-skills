The plan has unresolved contradictions around worktree isolation and execution-mode durability, and it marks state closed before later evaluation can still require code changes. ### F-003 [critical] ordering — design.md:69

**Evidence:**
```md
13. **Evaluation order is fixed.** All phase tasks `done` → evaluation agent (read-only, structured pass/fail) → only then phase-done `review-code --mode=both`. Evaluator never writes state; orchestrator logs dispositions. Evaluation blocker/critical re-dispatches a code-only fix agent (max 2) or stops for operator; does not silently mark phase done.
```

**Claim:** The plan closes all tasks before evaluation, but evaluation failures can trigger code changes without specifying how already-`done` task state and verifier evidence are reopened or refreshed.

**Impact:** A fix agent can modify product code after task closure, leaving done tasks with stale verifier evidence and stale reviewed commit ranges; phase-done can then certify code that was not covered by the recorded task-close evidence.

**Recommendation:** Add an explicit post-evaluation fix protocol: reopen affected tasks or create blocking follow-up tasks, rerun verifiers and complex-task reviews on the fix range, and only then allow phase-done.

**Confidence:** high

**Claim:** The plan requires reviewing a task commit range but never requires the claim report or orchestrator to prove an unambiguous per-task range or compute the destructive-diff signal from it.
