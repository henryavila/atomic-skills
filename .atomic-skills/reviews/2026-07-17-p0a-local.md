# P0-A recovery CLI — sealed local adversarial review

**Scope (read-only):** `src/recovery-cli.js`, `bin/cli.js`, `src/install.js` (repair + incomplete messaging), `src/uninstall.js` (force-incomplete), `tests/recovery-cli.test.js`  
**Pinned engine context:** `@henryavila/minimalist-installer` Driver still writes incomplete with **prior** effects only and flushes complete only at end (`node_modules/.../driver.js` ~62–103). No per-effect durable journal in the current pin.  
**Method:** file read + cross-check against engine recovery/lock/driver; no git history, no source edits.

---

## Summary

P0-A consumer half correctly **refuses pre-U silent resume** on `install --repair`, never writes `transaction: complete` on a partial tree, and routes wrong flags (`install --force-incomplete`, `uninstall --repair`). Residual recovery ledger + “do not hand-edit JSON” messaging exist and are tested at a smoke level.

The implementation is **not yet safe as an unblocking recovery product** under real dual-scope homes and concurrent install:

1. **Scope routing** can “succeed” against the wrong basePath (user) while project incomplete remains.
2. **`uninstall --force-incomplete` always exits 0** and prints “complete” even when reverse failed or incomplete was **retained** (post-U partial).
3. **pre-U force reverse of journaled (prior) effects** can tear down a previous good install while unjournaled mid-crash applies remain; residual ledger risk is a single enum without path inventory and without combining `partial_reverse`.
4. **Recovery locks (`~/.atomic-skills/locks`) do not serialize against Driver install locks (`~/.minimalist-installer/locks`)** — concurrent install vs force/repair is a real race.
5. **post-U trust heuristics are optimistic** relative to the pinned engine; a forged/premature `journalMode`/`journaledAt` path can reverse prior-only journals, clear incomplete, and claim `residualRisk: none` (false clean baseline — not a false `complete` marker, but operator-equivalent).

Tests cover pre-U refuse, force happy-path ledger, and mid-inject “no complete marker,” but miss dual-scope, exit codes on partial force, post-U residual honesty, and lock interop.

## Findings

### F1 — CRITICAL — Scope resolution can repair/force the wrong install root

- **Where:** `src/install.js:812–830`, `src/uninstall.js:79–104`, `bin/cli.js:77–84,112–117`
- **WHAT:**  
  - `install --repair` without `--project`: if a user manifest exists, scope is **always user**, even when the incomplete TX is only (or also) under the project Git root (`projectManifest && !userManifest` is the only auto-project case).  
  - `uninstall --force-incomplete` with `--yes` and both user+project installs: hard-defaults to **user** (`scope = 'user'`), never scans both for incomplete.
- **WHY:** Scope is chosen from “has any manifest” / install defaults, not from `describeRecovery` / incomplete state on each candidate base.
- **IMPACT:**  
  - Project incomplete + user complete → `install --repair` no-ops (“transaction is complete”) **exit 0**; project remains bricked.  
  - Dual incomplete → force only clears user; project stays incomplete; operator thinks recovery finished.  
  - This is a **CLI flag / routing false success**, not just UX.
- **RECOMMENDATION:**  
  1. Discover incomplete on **both** user and project bases.  
  2. If exactly one incomplete → target it.  
  3. If both → refuse with explicit paths unless `--project` / explicit scope.  
  4. Never report “nothing to repair” when the other scope is incomplete.  
  5. Add CLI tests for user-complete+project-incomplete and dual-incomplete.

### F2 — CRITICAL / HIGH — `force-incomplete` reports success and “complete” while reverse failed or incomplete retained

- **Where:** `src/recovery-cli.js:450–483`, `src/uninstall.js:120–128`
- **WHAT:**  
  - `forceIncompleteUninstall` always returns `exitCode: 0` and `ok: true` for incomplete paths, including when `failed.length > 0`.  
  - post-U partial reverse sets `clearIncomplete = false` (keeps incomplete) but still `exitCode: 0`.  
  - `uninstall.js` always prints `Force-incomplete complete (...)` after a non-throwing result.
- **WHY:** Mutator treats “ledger written / best-effort attempted” as operational success; outer CLI does not distinguish partial/retained incomplete.
- **IMPACT:**  
  - Scripts and operators treat exit 0 as unblocked reinstall readiness.  
  - post-U partial: incomplete still blocks install/uninstall, but message says complete.  
  - pre-U total reverse failure: incomplete cleared (`clearIncomplete` always true for pre-U) with failed effects — tree may still have journaled paths; residual honesty depends on operator reading the ledger, not the exit code.
- **RECOMMENDATION:**  
  - `exitCode: 1` when `failed.length > 0` or incomplete retained.  
  - Distinct actions: `forced-clean` | `forced-residual` | `partial-kept-incomplete`.  
  - Outer message must not say “complete” unless incomplete is gone **and** (pre-U residual ledger written | post-U failed empty).  
  - Tests for unknown effect + post-U partial exit codes.

### F3 — HIGH — pre-U force reverse of **prior** journal can delete too much of a prior good install

- **Where:** `src/recovery-cli.js:186–217, 408–450`; engine `driver.js:62–74` (prior effects on incomplete disk journal)
- **WHAT:** On the pinned pre-U engine, disk incomplete journals carry **previous complete install effects**, not effects applied after the incomplete marker. `forceIncompleteUninstall` still `reverseJournaledEffects` on that prior journal, then clears incomplete (pre-U always).
- **WHY:** Reverse is correct for “journaled ownership” but wrong as a mental model of “undo the crashed install.” For updates, crash mid-apply leaves:  
  - mutated new content (hash ≠ prior `installedHash` → reconcile revert **skips**),  
  - unmutated prior files (hash still matches → **deleted**),  
  - unjournaled new paths (remain).  
  Result is a **partially destroyed previous install** plus crash residue.
- **IMPACT:** Operator who expected “unblock + leave residual for reinstall” may lose still-valid prior files; reinstall then relies on adopt/clobber behavior. Opposite of “delete too little,” this is **delete the wrong generation**.
- **RECOMMENDATION:**  
  - Document explicitly that pre-U force is **destructive reverse of last durable journal**, not “abort in-flight update.”  
  - Prefer: clear incomplete + residual ledger **without** reversing prior effects when trust is pre-U and journal is known prior-only; leave full reverse to normal uninstall after a successful reinstall path — **or** reverse only after classifying first-install (empty/prior-empty) vs update.  
  - At minimum: ledger fields `journalShape: 'prior-effects-only'`, `destructiveReverse: true`, and warn before reverse when effectCount > 0 on pre-U.  
  - Tests: complete install → inject incomplete with prior effects + partial new files → force → assert which paths remain/deleted.

### F4 — HIGH — Residual ledger honesty gaps (risk enum, paths, combined failure modes)

- **Where:** `src/recovery-cli.js:295–319, 414–438`
- **WHAT:**  
  - pre-U force always sets `residualRisk: 'unjournaled_applies_possible'` even when `failed.length > 0` (masks `partial_reverse`).  
  - post-U repair success sets `residualRisk: 'none'` after reverse+`removeManifest` with **no disk scan** for leftover paths.  
  - Ledger lists failed effect types/ids but **not** unrecovered file paths (P0 plan: “residual recovery ledger of unrecovered paths”).  
  - `force` message says “You may reinstall now” whenever residualRisk ≠ `unjournaled_applies_possible` — including when incomplete was retained (post-U partial) if residualRisk is `partial_reverse`… actually message uses residualRisk === unjournaled for the pre-U branch; post-U partial still says “You may reinstall now” (`recovery-cli.js:479–481`) while incomplete remains.
- **WHY:** Risk is derived from trust flag + failed count only; success messaging not gated on incomplete clearance.
- **IMPACT:** Operators and automation cannot trust residualRisk / nextSteps as machine-readable recovery state; “reinstall now” is a **false clean** signal on post-U partial.
- **RECOMMENDATION:**  
  - Multi-flag risk: `{ unjournaledApplies, reverseFailures, incompleteRetained, outsideJournalRuntime }`.  
  - Gate “reinstall now” on `!incomplete && failed.length === 0` (pre-U may still warn unjournaled).  
  - Best-effort path inventory from journaled beforeState + optional walk of known install roots for leftovers.  
  - Never `residualRisk: 'none'` without either post-U proof or a clean filesystem probe.

### F5 — HIGH — Lock/race: recovery lock root ≠ Driver install lock root

- **Where:** `src/recovery-cli.js:243,374` + `src/runtime-locks.js:133–150`; engine `lock.js:23,143–148` + `driver.js:42,112`; `src/installer.js` does not pass `lockRoot`/`resourceIdentities`
- **WHAT:** Recovery mutators use `withSharedRuntimeLocks` → `~/.atomic-skills/locks` with identities `registry`, `runtime-index`, `install-root`. Driver install/uninstall uses `acquireInstallLocks` → default **`~/.minimalist-installer/locks`** with only `install-root` (unless config overrides — buildInstaller does not).
- **WHY:** Same identity string, **different lock directories** → no mutual exclusion.
- **IMPACT:** Concurrent `install` / `uninstall` / `--repair` / `--force-incomplete` can interleave: reverse while Driver applies, or clear incomplete while another process is mid-journal write → torn manifests, double reverse, or incomplete re-introduced under recovery’s feet.
- **RECOMMENDATION:**  
  - Pass shared `lockRoot` (`~/.atomic-skills/locks`) and the same identity set into `defineInstaller` / Driver **or** have recovery acquire engine locks on the engine root **and** install-root.  
  - Hold locks across the full “inspect → mutate → clear” critical section (already mostly true for mutators; install incomplete **check** at `install.js:852` is still outside any lock — TOCTOU before Driver).  
  - Test: overlapping child processes force vs installSkills.

### F6 — HIGH — post-U trust classification can authorize reverse+clear on prior-only journals (false clean baseline)

- **Where:** `src/recovery-cli.js:62–83, 286–330`
- **WHAT:** `classifyJournalTrust` returns post-U if **any** of:  
  - `tx.journalMode === 'per-effect'`  
  - `tx.durablePerEffect === true`  
  - `tx.journalTrust === 'post-U'`  
  - **or** every effect has `flushedAt` | `durable === true` | `journaledAt`  
  Current pin never writes these; speculative path reverses all journaled effects and on full reverse success **removes manifest** with `residualRisk: 'none'`.
- **WHY:** Trust is declarative flags / weak markers, not verification that disk journal equals applied set. `journaledAt` especially does not prove durable per-effect flush. Empty-effects + `journalMode` alone is post-U with zero reverse work then clear.
- **IMPACT:** Mis-pin, hand-edit, or premature flag → treat pre-U prior-only journal as post-U → reverse prior install, clear incomplete, claim clean → **unjournaled mid-crash files remain** with no incomplete gate. This is the review focus “false complete” family: not writing `state: complete`, but **false installable/clean marker removal**.
- **RECOMMENDATION:**  
  - Until engine pin actually flushes per effect, **hard-disable post-U mutator** (always refuse like pre-U) or require package semver/capability bit from installer.  
  - Do not treat `journaledAt` alone as durable.  
  - Require non-empty applied evidence + capability flag.  
  - On clear, default residualRisk to at least `unverified` unless a probe passes.  
  - Tests: forged `journalMode` on prior-only incomplete must not clear with residualRisk none.

### F7 — MEDIUM — Mid-repair crash window after reverse, before ledger

- **Where:** `src/recovery-cli.js:288–320` (repair), analogous force `406–443`
- **WHAT:** Order is reverse → (inject seam) → write ledger → clear incomplete. Crash/kill after reverse and before ledger leaves: files already reverted, incomplete still present, **no residual ledger**.
- **WHY:** Ledger is not written until after all reverts complete; no staging “recovery-in-progress” record before first destructive revert.
- **IMPACT:** Re-entry must re-reverse (mostly idempotent for missing files) but operator loses discoverability promised by “mid-repair leaves incomplete and/or ledger.” Tests only inject at `classify` / `before-reverse`, not `after-reverse`.
- **RECOMMENDATION:** Write a `recovery-in-progress` ledger (effect list + intent) **before** first revert; update after each effect; tests with `injectFailAfter: 'after-reverse'`.

### F8 — MEDIUM — pre-U silent resume: refused on repair, but force clear is an implicit “resume gate open” without tree consistency

- **Where:** `src/recovery-cli.js:268–283` (good refuse); `450–459` (pre-U always clear); `src/install.js:852–857` (fail-closed)
- **WHAT:** Silent resume of incomplete journal is correctly refused on `--repair`. However pre-U force **always** unlinks incomplete after ledger write even when reverse fully failed, then install can proceed over a dirty tree (adopt path may claim leftovers).
- **WHY:** Design prioritizes unblocking over journal↔disk consistency (allowed by C-only exit gate, but easy to misread as recovered).
- **IMPACT:** Not silent resume of the same TX, but **silent permission to install atop residue** after a failed reverse — related to residual honesty (F2/F4).
- **RECOMMENDATION:** If `failed.length > 0`, keep incomplete **or** require `--force-incomplete --i-know` second flag to clear. Default clear only when reverse had zero failures.

### F9 — MEDIUM — CLI / messaging routing nits

- **Where:** `bin/cli.js:71–74,108–110` (good cross-flag reject); `src/install.js:838–841`; `src/uninstall.js:117`
- **WHAT:**  
  - Catch paths call `incompleteOperatorMessage()` **without trust** (defaults pre-U) even mid post-U repair.  
  - `install --repair` success path tells re-run install but does not chain reinstall (strategy b half-implemented — documented, OK if intentional).  
  - No detection when `--repair` no-ops complete on user while project incomplete (see F1).  
  - `forceIncomplete` with no manifests at all defaults scope user and no-ops — fine; if project has incomplete but `readManifest` throws (invalid JSON), local `src/manifest.js` **throws** on parse while engine `inspectTransaction` treats invalid JSON as incomplete — **scope discovery may crash before force runs**.
- **WHY:** `uninstall.js` / install scope use `readManifest` from `src/manifest.js` (JSON.parse uncaught); recovery uses engine `describeRecovery` (tolerant).
- **IMPACT:** Corrupt incomplete manifest can make `uninstall --force-incomplete` throw during scope detection before mutator; operator still stuck.
- **RECOMMENDATION:** Scope probes should use tolerant inspect (or try/catch parse) consistent with recovery; pass trust into catch messages; optionally auto-reinstall after successful post-U reverse under same lock.

### F10 — LOW / MEDIUM — Test gaps vs claimed P0-A gates

- **Where:** `tests/recovery-cli.test.js` (whole file)
- **WHAT:** Covered well: pre-U refuse, force reverse+ledger, mid-inject no `complete` marker, CLI smoke for refuse/block/force.
