## Summary
The fork hierarchy rule has correctness gaps in selection and scoping. It can still pick the parent while clearing ambiguity when branch precedence narrows the pool before collapse, and it can collapse unrelated plans across projects because fork relationships are keyed only by slug.

The new sidecar reads are also fail-open at the wrong boundary: malformed `links.json` now aborts focus emission and reconciliation entirely. Cyclic fork edges are silently treated as a resolved hierarchy, which clears the ambiguity marker for corrupt graph state.

## Findings

### F-001 [major] correctness — scripts/emit-focus.js:151-154

**Evidence:**
```js
  const superseded = supersededParentSlugs(claimers);
  const effClaimers = claimers.filter((p) => !superseded.has(p.fm?.slug));
  const effPool = pool.filter((p) => !superseded.has(p.fm?.slug));
  const plan = recentFirst(effPool.length ? effPool : pool)[0];
```

**Claim:** A fork child can be a claimer but excluded from `pool`, causing the superseded parent to be resurrected and selected while `multipleActivePlans` is cleared.

**Impact:** On a branch where the parent has an exact `branch:` match and the active child is unbranched, `claimers` contains both parent and child, `pool` contains only the exact-match parent, `effPool` becomes empty, and the fallback selects the parent. The digest then reports focus on the parent and `multipleActivePlans=false`, hiding the active child that should own the hierarchy.

**Recommendation:** Apply fork supersession before constructing the selection pool, or rebuild `pool` from `effClaimers` using the same exact-branch/unbranched precedence so a surviving child claimer remains selectable. Add a test for exact-branch parent plus unbranched fork child.

**Confidence:** high

### F-003 [major] correctness — scripts/emit-focus.js:126-153

**Evidence:**
```js
function supersededParentSlugs(plans) {
  const present = new Set(plans.map((p) => p.fm?.slug).filter(Boolean));
  const superseded = new Set();
  for (const p of plans) {
    const parentSlug = p.spawnedFrom?.plan;
    if (parentSlug && present.has(parentSlug)) superseded.add(parentSlug);
  }
  return superseded;
}
```

**Claim:** Fork supersession is keyed only by plan slug, so a child in one project can supersede an unrelated active plan with the same slug in another project.

**Impact:** In a multi-project `.atomic-skills/projects/<id>/...` tree, `projects/a/child` with `spawnedFrom.plan: parent` causes every active claimer whose `fm.slug` is `parent` to be removed, including `projects/b/parent`. Focus can resolve to the wrong project’s child and clear `multipleActivePlans` for unrelated work.

**Recommendation:** Key presence and supersession by project plus slug, using the child plan’s `projId` to resolve its parent within the same project. Add a multi-project duplicate-slug test for `buildFocusDigest`.

**Confidence:** high

### F-004 [major] correctness — scripts/reconcile-focus.js:160-165

**Evidence:**
```js
function collectForkDeferrals(planEntries) {
  const map = new Map();
  for (const { fm, spawnedFrom } of planEntries) {
    if (!fm || fm.status !== 'active' || !spawnedFrom?.plan || !spawnedFrom?.phaseId) continue;
    if (!map.has(spawnedFrom.plan)) map.set(spawnedFrom.plan, new Set());
    map.get(spawnedFrom.plan).add(spawnedFrom.phaseId);
  }
  return map;
}
```

**Claim:** Reconcile deferrals are keyed only by parent slug, so an active fork child can clear `current` on unrelated parents in other projects with the same slug and phase id.

**Impact:** If two projects both have an active plan slug `parent` with current phase `F3`, and only one project has an active child spawned from `parent/F3`, `reconcileDir` defers the `current` marker for both parents. The unrelated project loses its AGORA marker even though it has no fork child.

**Recommendation:** Include `projId` in `planEntries` and key deferrals by `{projId, parentSlug}` rather than slug alone. Add a reconcile test with two projects containing the same parent slug and phase id.

**Confidence:** high

  for (const p of plans) {

**Claim:** Cyclic fork edges can supersede every claimer, after which the resolver falls back to the original pool but reports no ambiguity.

**Recommendation:** Detect cycles and all-superseded collapses before clearing ambiguity. For invalid graphs, ignore the collapse or set `multipleActivePlans=true` and surface an error. Add tests for `A↔B` and self-referential `spawnedFrom.plan`.
