## Summary
The procedure introduces a fork-resume transaction but does not wire the archive gate tightly enough before the archive mutation, leaves required canonical-resolution inputs unspecified, and does not make marker durability robust for write exceptions or partial pause-mode writes. The new tests mostly exercise locally defined helper behavior rather than the actual fork-resume decision/transaction flow, so they would pass with the procedural logic absent or miswired.

## Findings

### F-001 [major] API contract — skills/shared/project-assets/project-transitions.md:236

**Evidence:**
```md
   - **accept** → resolve the parent's canonical dir (`resolveCanonicalParentDir`), then call `writebackOrDefer({ canonicalFile: <parent plan.md>, childPlanDir, readToken, mutate: <resume mutation>, pending: { target: 'parent-plan', parent: <plan>, op: 'resumeParent', args: { phaseId }, readToken, detectedAt: <now> } })`.
```

**Claim:** The procedure calls `resolveCanonicalParentDir` without specifying the required `parentSlug`, `parentBranch`, `projectId`, and `worktrees` inputs, and the `spawnedFrom` edge only supplies `{ plan, phaseId, mode }`.

**Impact:** An agent executing this cannot deterministically locate the canonical parent plan in parallel mode; it may guess a branch, write to the child worktree copy, or abort before recording/resuming the parent.

**Recommendation:** Specify the exact resolution sequence and call shape: obtain `projectId` from the child plan context, obtain `parentBranch` from a stored fork-time field or a documented parent plan descriptor source, parse `git worktree list --porcelain` with `parseWorktrees`, call `resolveCanonicalParentDir({ parentSlug: plan, parentBranch, projectId, worktrees })`, set `canonicalFile = join(parentDir, 'plan.md')`, and abort before archive finalization if any input is missing.

**Confidence:** high

### F-002 [major] procedure ordering — skills/shared/project-assets/project-transitions.md:213-215

**Evidence:**
```md
2. **Fork-link resume offer (read the edge BEFORE any finalize)**: when the target is a plan, read its child→parent edge with `getSpawnedFrom(<target-plan-dir>)` (`src/links-sidecar.js`) **before** touching the plan's status or moving any file. When the edge is present (`{ plan, phaseId, mode }`) the target is a fork child of plan `plan` at anchor phase `phaseId` (degrau 7.5 — the `spawnedFrom` edge written at fork time): **offer** to resume the parent at that anchor — the offer is opt-in, printed for the user, and is **NEVER applied automatically**. When `getSpawnedFrom` returns `null`, skip this step silently — a non-forked plan archives exactly as it did before. Reading the edge here, ahead of the finalize in steps 3–4, is what lets the `fork-resume` application keep its transaction order (the parent writeback precedes the child-archive finalize). The deterministic application of this offer across both modes (`pause` / `parallel`) and its edge cases (accept / refuse / no-TTY / failed writeback) is the **`fork-resume`** step below.
3. **Plan archival**:
   - Set the plan's `status: archived`.
```

**Claim:** The archive procedure reads and offers fork-resume, but the next numbered step archives the plan without an explicit branch that executes `fork-resume` and returns before step 3 on refuse, no-TTY, or failed writeback.

**Impact:** A step-by-step agent can correctly perform the offer and then continue into `status: archived`/move operations, producing exactly the forbidden state: a child finalized while parent resume is declined, deferred, failed, or not yet durably recorded.

**Recommendation:** Make archive step 2 a hard gate: if `spawnedFrom` exists, execute `fork-resume`; proceed to archive steps 3-4 only on accepted successful parent resume; on refuse/no-TTY/failure, record `pendingWriteback` and terminate the archive command before any status or move mutation.

**Confidence:** high

### F-003 [major] error handling — skills/shared/project-assets/project-transitions.md:236-241

**Evidence:**
```md
   - **accept** → resolve the parent's canonical dir (`resolveCanonicalParentDir`), then call `writebackOrDefer({ canonicalFile: <parent plan.md>, childPlanDir, readToken, mutate: <resume mutation>, pending: { target: 'parent-plan', parent: <plan>, op: 'resumeParent', args: { phaseId }, readToken, detectedAt: <now> } })`. On `{ ok: true }` the parent is resumed and any stale marker cleared → continue to `archive` steps 3–4 (finalize the child). On `{ ok: false, conflict, deferred }` the parent moved since the read — the call has **already** recorded the durable `pendingWriteback` (it never leaves recording to the caller); **abort the archive finalize**, surface the conflict + recovery, leave the child `active`/un-archived.
...
   - **accept** → apply the resume mutation directly to the parent plan.md + anchor initiative file (single writer, same tree — no CAS), then run `node scripts/refresh-state.js` to cascade focus markers + the `focus.json` digest. Continue to `archive` steps 3–4.
...
   - A **write failure** on accept is treated like the parallel conflict: the durable marker stands and the child does not finalize until recovery.
```

**Claim:** The failure path only covers returned `{ ok: false }` conflicts and asserts durability for pause-mode write failures without specifying marker-before-mutation ordering or exception handling.

**Impact:** If `mutate`, `writeFileSync`, `renameSync`, the direct parent plan write, the anchor initiative write, or `refresh-state` throws, the procedure can exit with no `pendingWriteback`; in pause mode it can also leave parent plan and anchor initiative partially updated with no reliable recovery marker.

**Recommendation:** Define failure handling explicitly: create or preserve the child `pendingWriteback` before any non-atomic pause-mode parent mutation and clear it only after both parent files and `refresh-state` succeed; wrap parallel `writebackOrDefer` exceptions to record the same marker before aborting archive finalization.

**Confidence:** medium

### F-002 [major] error handling — skills/shared/project-assets/project-transitions.md:239

**Evidence:**
```md
3. **mode `pause`** (parent in the same tree, paused by the fork):
   - **accept** → apply the resume mutation directly to the parent plan.md + anchor initiative file (single writer, same tree — no CAS), then run `node scripts/refresh-state.js` to cascade focus markers + the `focus.json` digest. Continue to `archive` steps 3–4.
```

**Claim:** The pause-mode accept path says the durable marker “stands” on write failure, but never records it before the direct parent plan and anchor initiative writes that can fail.

**Impact:** If the direct write fails before a marker is created, recovery has no declarative `resumeParent` operation to replay; if one parent file is written and the other fails, the parent can be left partially resumed with no guaranteed child-side recovery marker.

**Recommendation:** Specify pause accept as: record `pendingWriteback` before any parent mutation; perform both parent plan and anchor initiative updates plus `refresh-state`; clear the marker only after all writes succeed; on any exception, leave the marker and abort archive finalization.
